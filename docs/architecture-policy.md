# Архитектурная политика CascadeIDE

**Статус:** действующая; новый код и крупные доработки следуют ей; существующий код мигрирует постепенно.  
**Связь:** дополняет [git-and-submodules-v1.md](git-and-submodules-v1.md), [MCP-PROTOCOL.md](MCP-PROTOCOL.md), [debug-human-agent-parity-v1.md](debug-human-agent-parity-v1.md) и общие правила репозитория.

## Цель

Сохранить скорость разработки **одного** десктопного приложения (Avalonia + MVVM) и при этом:

- не раздувать один «God ViewModel» без необходимости;
- отделять **UI-состояние**, **сценарии приложения** и **доступ к внешнему миру** (диск, процессы, git, сеть);
- упрощать тесты и ревью за счёт **узких** классов и явных границ;
- по отладке стремиться к **единому слою** для человека и агента (брейкпоинты, останов, стек/переменные) — см. [debug-human-agent-parity-v1.md](debug-human-agent-parity-v1.md).

Политика — **прагматичная**: не полный DDD и не обязательный hexagonal на весь код; там, где даёт окупаемость.

## Выбранный стиль: слои + вертикальные срезы

### Горизонтально (слои)

| Слой | Назначение | Примеры в проекте |
|------|------------|-------------------|
| **UI** | Views (AXAML), code-behind только для чисто view-задач | `Views/*.axaml` |
| **Presentation** | ViewModels, команды, биндинги | `ViewModels/` |
| **Application / domain logic** | Сценарии без привязки к Avalonia: парсеры, координация | `Services/SolutionParser`, оркестрация |
| **Infrastructure** | Файлы, процессы, git, HTTP, MCP-транспорт | `Services/` (часть), `IdeMcpServer` |

**Правило:** ViewModel **не** содержит прямых деталей запуска процессов/чтения файлов, если это можно отдать **сервису** с интерфейсом (или статическому helper’у с чётким именем и одной ответственностью — для совсем тонкой логики).

### Вертикально (срезы / фичи)

Крупная возможность (**панель Git, терминал, отладка, чат**) оформляется как **срез**:

- общая идея: `Features/<Имя>/` **или** согласованное с репо размещение (`Panels/`, `Git/` — главное **единообразие**);
- в срезе: ViewModel панели, при необходимости — модели только этой фичи, **тонкие** обращения к общим сервисам.

**Правило:** новая **нижняя/боковая панель** или крупный самостоятельный блок → **отдельный `*ViewModel`**, по возможности отдельный `*View`. `MainWindowViewModel` остаётся **композитором**: создаёт дочерние VM, пробрасывает зависимости, реагирует на события уровня окна (смена решения, закрытие и т.д.).

**Каталог срезов:** `Features/<Имя>/` (неймспейс `CascadeIDE.Features.<Имя>`), см. [Features/README.md](../Features/README.md). Пошаговый перенос из монолитного VM — [architecture-migration.md](architecture-migration.md).

## События, UI-поток и нагрузка (текущий план)

**Зачем:** в IDE много источников и потребителей сигналов (сборка, диагностики, MCP, LSP и т.д.). Без явных правил растёт связность, случайные вызовы между подсистемами и ошибки потоков.

**Порядок внедрения (strangler, без обязательной смены всего кода сразу):**

1. **Границы и уведомления** — источники по возможности только **публикуют факты** (событие, запись в шину, вызов узкого контракта); потребители **не тянут** друг друга напрямую там, где достаточно реакции на факт. Конкретная механика (тонкий event bus, `IObservable`, медиатор) выбирается по месту; цель — **слабая связность** и предсказуемый граф зависимостей.
2. **Политика UI-потока** — обновления ViewModel и UI после работы в фоне проходят через **одну согласованную точку** маршалинга: **`IUiScheduler`** и **`UiScheduler.Default`** (реализация — `AvaloniaUiScheduler`, внутри — `Dispatcher.UIThread`), без размазанного `Post`/`Invoke` по коду.
3. **Очереди и батчинг (точечно)** — там, где поток данных **перегружает** UI (длинный вывод сборки, лавина диагностик), допускаются канал/очередь и **пакетное** обновление VM, чтобы не дергать привязки на каждую строку.

**Связь с паттернами:** listener/pub-sub — для топологии «много ко многим»; producer/consumer — для участков с реальной нагрузкой; отдельный «dispatcher» слой — в первую очередь **маршалинг на UI-поток** и при необходимости маршрутизация команд, а не замена DI.

Статус по шагам — в [architecture-migration.md](architecture-migration.md) (фаза 5).

## Отложенные идеи расширяемости

- **MEF / динамические плагины (загрузка DLL из каталога):** **не** целевой шаг для ближайшего рефакторинга. Имеет смысл, когда появится продуктовая цель «сторонние расширения» или обязательная runtime-композиция из папки. До этого модули остаются **проектами в решении** + явная регистрация в DI / композиции приложения.

## Правила по конкретным артефактам

1. **`MainWindowViewModel`**  
   - Допустимо держать композицию и мост к MCP.  
   - **Не** добавлять сюда объёмную логику новой фичи (десятки методов git, парсинг путей, поиск решений) — вынос в панельный VM + сервисы.

2. **Модели списков** (`Models/GitStatusRow` и т.п.)  
   - Данные и флаги для отображения; без тяжёлой бизнес-логики.

3. **Внешний мир**  
   - Git, `Process`, длительные операции — за **абстракцией**, которую в тестах можно подменить (интерфейс или `internal` + `InternalsVisibleTo` для тестов — по ситуации).

4. **MCP**  
   - Контракты агента (`IIdeMcpActions`, DTO) — **стабильные**; реализация делегирует в сервисы/VM без дублирования бизнес-правил в каждом тули.

5. **Именование и размещение**  
   - Один новый модуль — один **понятный** корень в неймспейсе (`CascadeIDE.Features.Git` или как зафиксируем в первом же срезе).

## Миграция (strangler)

- **Новый код** — сразу по политике.  
- **Старый код** в монолитном VM — трогаем при доработке фичи или при явном рефакторинге; полный перенос не обязателен по дедлайну.

## Когда отклоняться

Короткий прототип или spike — допускается упрощение; перед merge в основную ветку либо приводим к политике, либо помечаем в PR и создаём задачу на вынос.

## Версионирование документа

- **v1** — зафиксированы слои, срезы, роли `MainWindowViewModel` и внешних сервисов.  
- **v1.1** — целевой каталог `Features/`, ссылка на [architecture-migration.md](architecture-migration.md); инфраструктура git через `IGitCommandRunner`.  
- **v1.2** — план: события, UI-поток, точечные очереди; MEF/плагины — отложенная идея.  
- Изменения политики — отдельным коммитом с обновлением этого файла и при необходимости ADR в [docs/adr/README.md](adr/README.md).
