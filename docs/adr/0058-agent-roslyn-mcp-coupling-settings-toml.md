# ADR 0058: Сопряжение агента и Roslyn MCP в `settings.toml` (лимиты, виды узлов, таймауты, пресеты)

**Статус:** Proposed  
**Дата:** 2026-04-18  
**Обновлено:** 2026-04-18 — [§6](#adr0058-p6): именованные профили в TOML, Manager переключения, режимы Auto-Focus / Combat / Glide Slope; ось ортогональна UI-режимам [0010](0010-ui-modes-toml-configuration.md).  
**Обновлено:** 2026-04-18 — [§7](#adr0058-p7): третий монитор как стратегический EFB (`Profile.GlobalMap`) vs тактический PFD; инвариант частоты обновления.

**Связь:** [0008](0008-mcp-contracts-and-testable-infrastructure.md) (контракты MCP и тестируемая инфраструктура), [0010](0010-ui-modes-toml-configuration.md) (UI-режимы `Focus` / `Balanced` / `Power` / `Flight` и т.д. — **отдельная ось** от профилей сопряжения ниже), [0017](0017-multi-window-workspace-and-agent-surfaces.md) (несколько окон / мониторов, `presentation`, MFD), [0021](0021-pfd-mfd-cockpit-attention-model.md) (PFD / MFD, масштабы внимания), [0028](0028-user-settings-toml-localappdata-and-secrets.md) (`settings.toml`, `%LocalAppData%\CascadeIDE\`), [0039](0039-workspace-navigation-affordances.md) (навигация, пресеты MCP, `kind_filter`, subgraph; Semantic Map в UI), [0040](0040-lsp-launch-line-settings-toml-presets-and-environment.md) (паттерн: пресеты и параметры в TOML без пересборки), [0051](0051-intent-based-attention-routing-toml.md) (маршрутизация внимания по intent — смежно с «намерением» переключить профиль), [0053](0053-semantic-map-control-flow-pfd.md) (Semantic Map, control flow на PFD).

---

## Контекст

Агентский слой (LLM + оркестрация tool calls) и **Roslyn** как источник семантики связаны через **MCP-инструменты** (внешний процесс `roslyn-mcp` или эквивалент). Без явных правил «сколько и чего отдавать» агент:

- получает **слишком плотный** ответ (токены, задержка, шум в контексте);
- или наоборот **недополучает** структуру, нужную для задачи (рефакторинг vs обзор).

Пользователь должен иметь возможность **менять поведение сопряжения без правки C#** в IDE и без смены системного промпта: каноничное место для такого в продукте уже задаётся **[0028](0028-user-settings-toml-localappdata-and-secrets.md)** — **`settings.toml`**.

**Интуиция (не норма ADR):** в образе «авиационной системы» агентский слой сопоставим с **бортовым контуром запросов**, Roslyn MCP — с **датчиком/шлюзом семантики**; настройки ниже — **протокол обмена** (объём, фильтр, тайминг), а не дублирование логики Roslyn в IDE.

**Уже есть в продукте (не отменяется этим ADR):**

- `[semantic_map]` — **вид и глубина** отображения Semantic Map в UI (`view`, `depth`) — см. `SemanticMapSettings`, [0039](0039-workspace-navigation-affordances.md), [0053](0053-semantic-map-control-flow-pfd.md).
- Пресеты навигации и параметры вызова MCP **`get_workspace_navigation_context`** — [0039 § Agent/MCP](0039-workspace-navigation-affordances.md#adr0039-mcp-workspace-navigation).

Этот ADR вводит **отдельный слой**: параметры **именно сопряжения агент ↔ Roslyn MCP** (ограничение выдачи, виды узлов, ожидание компиляции, именованные пресеты запросов), чтобы не смешивать их с чисто **UI-настройками** карты и с **одноразовыми** аргументами конкретного вызова тула.

---

## Решение (принципы)

<a id="adr0058-p1"></a>

### 1. Одна декларативная схема — несколько потребителей

Настройки читают **только те компоненты, для которых ключ нормативен**. Минимальная таблица ответственности (уточняется при реализации):

| Категория | Пример смысла | Кто обязан учитывать |
|-----------|----------------|----------------------|
| Лимиты объёма ответа | `max_nodes_per_query`, лимиты глубины обхода | Процесс **Roslyn MCP** (или адаптер, агрегирующий граф до ответа) |
| Фильтр видов узлов / символов | `included_kinds` / исключения | **Roslyn MCP** при построении projection |
| Таймауты / «грязная» семантика | ожидание полной компиляции vs stale graph | **Roslyn MCP** + при необходимости **IDE** (если запрос идёт через hosted workspace) |
| Пресеты режимов запроса | `ExploreMode` vs `RefactoringMode` (имена условные) | **Roslyn MCP** (разворачивает пресет в набор параметров по умолчанию); **агент/хост** может переопределить в конкретном вызове |

**Правило:** не дублировать один и тот же смысл в трёх местах (TOML IDE, env, хардкод сервера) без явного **порядка приоритета** (например: аргумент вызова > секция TOML > дефолт сервера) — зафиксировать в реализации и в [MCP-PROTOCOL.md](../MCP-PROTOCOL.md) при появлении полей.

<a id="adr0058-p2"></a>

### 2. Четыре группы параметров (контрактные оси)

1. **Throttling & scoping** — ограничения на объём и глубину выдачи семантики за один запрос (чтобы снизить перегруз контекста агента и стоимость ответа). Примеры имён: `max_nodes_per_query`, `max_recursion_depth` (или эквиваленты под конкретный API графа — уточняется в спецификации тула).

2. **Маппинг видимости (capabilities)** — декларативный список **какие классы узлов/символов** участвуют в выдаче (например только объявления методов/свойств; поля `Field` отключены). Семантика: **контракт видимости**, а не дублирование всего дерева Roslyn в конфиге.

3. **Таймауты и режимы согласованности** — ждать ли завершения фоновой компиляции; допустим ли ответ по **устаревшей** семантической модели; верхние границы времени на тяжёлые запросы (в т.ч. к **Semantic Map v1** и родственным тулам).

4. **Инструментальные пресеты** — именованные наборы: что включает режим вроде `controlFlow` / `architecture` / «рефакторинг» (расширенные usages) vs «обзор» (только крупная структура). Связь с уже существующими **пресетами навигации** [0039](0039-workspace-navigation-affordances.md): либо **ссылка по имени**, либо **ортогональные** пресеты Roslyn MCP — решение фиксируется в спецификации, без неявного слияния двух разных механизмов.

<a id="adr0058-p3"></a>

### 3. Размещение в TOML (целевая секция)

Предлагается выделить явную секцию верхнего уровня, например **`[agent.roslyn_mcp]`** (или эквивалент согласованный с парсером вложенных таблиц в CascadeIDE), чтобы:

- не смешивать с `[semantic_map]` (UI);
- не смешивать с `[[navigation.presets]]` без явного маппинга.

Точные ключи и значения по умолчанию — **в отдельном чертеже или в приложении к этому ADR** после прототипа; в статусе **Proposed** этот ADR фиксирует **оси и границы**, не обязательно финальные имена полей.

<a id="adr0058-p4"></a>

### 4. Версионирование и обратная совместимость

- Добавление новых опциональных ключей — **несовместимо по смыслу** только если меняется семантика по умолчанию; предпочтительно **явные дефолты** в документации и в коде десериализации.
- Смена обязательной формы секции — bump схемы настроек (как в других слоях [0028](0028-user-settings-toml-localappdata-and-secrets.md) / политика миграций).

<a id="adr0058-p5"></a>

### 5. Минимальный v0 vs отложенное

**v0 (когда возьмут в работу):** лимиты объёма + один-два таймаута + один именованный пресет «обзор vs глубже» — без полного маппинга всех Roslyn kinds.

**Отложено до необходимости:** исчерпывающий справочник `included_kinds`, сложные правила recursion по вызовам, объединение всех пресетов навигации и Roslyn в один мегаконфиг, **полная** автоматика переключения профилей по эвристикам (см. [§6](#adr0058-p6) — допускается поэтапно).

<a id="adr0058-p6"></a>

### 6. Несколько профилей в TOML, менеджер переключения и режимы работы

Помимо «плоского» набора ключей в `[agent.roslyn_mcp]`, допускается (и по продуктовому замыслу предполагается) **несколько именованных профилей сопряжения** — целостных наборов параметров (лимиты, `depth`, фильтры kinds, таймауты, пресеты запросов к MCP), **заранее описанных и проверяемых** в TOML. Имена в духе **`Profile.Flight`**, **`Profile.Debug`**, **`Profile.DeepScan`** — **условные иллюстрации**: в реализации они маппятся на таблицы вроде `[[agent.roslyn_mcp.profiles]]` с полем `id` (точная схема — при внедрении).

**Инвариант:** переключение активного профиля — это **выбор одного из заранее валидных наборов правил** для контракта с Roslyn MCP, а не подмена произвольных ключей «на лету» как при агрессивном hot-reload всего конфига. Слой **Manager** (рабочее имя) в IDE:

- держит ссылку на **текущий профиль**;
- применяет его к запросам в сторону Roslyn MCP и к согласованному слою Semantic Map / PFD ([0053](0053-semantic-map-control-flow-pfd.md));
- не генерирует недетерминированные комбинации параметров в обход TOML.

**Ось UI [0010](0010-ui-modes-toml-configuration.md)** (`workspace.mode`, семейства `Flight` / `Editor` в bundle и т.д.) и **ось профилей сопряжения** — **разные**. Допустимы: (a) **независимое** переключение, (b) **политика маппинга** «при смене UI-режима подставляй профиль по умолчанию», (c) явная таблица соответствий в TOML. Без явной политики **не** считать `Profile.Flight` синонимом строки `mode = "Flight"` в workspace — имена могут совпадать только по смысловому соглашению команды.

Переключение профиля — **по намерению** и/или **автоматически** по сигналам среды (ниже — целевые сценарии; детали эвристик и антидребезга — в реализации и тестах).

| Режим (рабочие имена) | Триггер (идея) | Эффект на контракт с Roslyn / карту |
|------------------------|----------------|-------------------------------------|
| **Auto-Focus** | Движение курсора / смена активного символа в **Forward** (редактор) | Сужение контракта: «покажи семантику **вокруг текущего метода**» — соседи по вызовам/контракту в пределах заданного scoping, без раздувания графа всего решения. Жёсткий контракт выдачи для снижения шума. |
| **Combat Mode** | Roslyn сообщает **ошибку** в релевантном блоке (диагностика в зоне внимания) | Автоматическое **расширение** контракта (например `depth++`, дополнительные kinds или рёбра зависимостей), чтобы искать причину **в зависимостях**. На PFD — **дополнительные связи**, которые в спокойном профиле скрыты политикой declutter ([0055](0055-skia-instrument-composition-pipeline.md)). |
| **Glide Slope** | Спокойный ввод текста (новый код без активной диагностической «войны») | Режим **минимального шума**: реже обновлять граф, не «пульсировать» Semantic Map на каждый keystroke; приоритет стабильности картины внимания над полнотой. |

Связь с **намерением** в смысле routing ([0051](0051-intent-based-attention-routing-toml.md)): явное переключение профиля может приходить из UI-команды, MCP (`ide_execute_command`) или политики «intent → профиль» — при появлении продукта отдельно зафиксировать, чтобы не плодить скрытые магические связи.

<a id="adr0058-p7"></a>

### 7. Третий монитор: навигационный планшет (EFB) и «стратегический» профиль

**Намерение:** при наличии **третьего монитора** вынести на него один из профилей сопряжения в **статичном** режиме — по смыслу ближе к **Electronic Flight Bag (EFB)** / боковому планшету: **другой масштаб времени и пространства**, чем у основного **PFD** в центре.

В авиации пилот смотрит на PFD **очень часто** (текущий курс, режим), а на **карту всего маршрута** на отдельном дисплее — **реже**, раз в несколько минут или по фазе полёта. Аналог для CascadeIDE:

| Зона | Роль | Профиль сопряжения (идея) | Частота обновления |
|------|------|---------------------------|--------------------|
| **Центр (Forward / PFD)** | **Тактический** режим | Динамический профиль (например узкий контракт MCP, Auto-Focus / control flow в [SkiaHost](0053-semantic-map-control-flow-pfd.md)): граф **подстраивается под курсор** — вошла в метод, увидела `controlFlow`, вышла — сузилось. «Радар ближнего боя». | Высокая (события редактора, смена символа; с политиками Glide / debounce). |
| **Третий монитор (MFD / отдельный TopLevel)** | **Стратегический** режим | Статичный профиль, условно **`Profile.GlobalMap`**: **глобальная** или **слойная** архитектура модуля — «скелет» решения. | **Низкая:** не дергать от каждого движения курсора. Обновление при **фоновой компиляции** (изменилась структура проекта / графа) или по **явному запросу** пользователя («обновить карту»). |

**Инвариант:** экран на третьем мониторе **не обязан** следовать тактическому ритму центра. Он держит **пространственную ориентацию**: когда ты глубоко в рефакторинге метода на PFD, взгляд на MFD возвращает контекст — «я всё ещё в этом слое / за границе интерфейса не выхожу». Это **не** дублирование того же Skia-состояния с двумя частотами; это **сознательное** сопоставление **поверхности** ([0017](0017-multi-window-workspace-and-agent-surfaces.md) — второй `TopLevel`, `presentation`) и **профиля** Roslyn MCP.

**Связь с [0021](0021-pfd-mfd-cockpit-attention-model.md):** MFD как зона **вторичного** контура — согласуется с идеей «карта боя» vs «шкаф»; здесь MFD дополнительно получает **семантику профиля** (стратегический слой, не тактический).

**Открытый момент реализации:** привязка «профиль X → окно/поверхность Y» в TOML или конфиге workspace ([0017](0017-multi-window-workspace-and-agent-surfaces.md), [0050](0050-declarative-instrument-zone-placement-toml.md)) — чтобы не хардкодить только `GlobalMap` на третий монитор у одного пользователя.

## Последствия

- Появляется **явный контракт** для обсуждения PR: «эта настройка относится к MCP-серверу Roslyn или к IDE».
- Тесты: контрактные сценарии на **дефолты** и на **merge** приоритетов (аргумент вызова vs файл).
- Документация пользователя: короткий подраздел в User Guide / справке, когда слой будет реализован.

---

## Отклонённые / отложенные альтернативы

- **Только переменные окружения** — хуже воспроизводимости и шаринга профиля «как у меня в IDE».
- **Только системный промпт агента** — не даёт детерминированного поведения тула и паритета человек/агент на уровне процесса.

---

## Открытые вопросы

- Нужна ли **синхронизация** части ключей с конфигом **standalone** `roslyn-mcp` (если агент вызывает сервер вне CascadeIDE) — одна схема или две с явным маппингом.
- Какие тулы Roslyn MCP входят в scope первой реализации (только workspace navigation / semantic map / оба).
- **Combat Mode:** гистерезис и debounce (когда ошибка «пропала», как быстро возвращаться к узкому профилю) — чтобы не мигать графом на границе чистой/грязной сборки.
- **Glide Slope vs Auto-Focus:** не конфликтуют ли эвристики (движение курсора одновременно с набором текста) — приоритеты слияния сигналов.
- Таблица **соответствия** UI-режима [0010](0010-ui-modes-toml-configuration.md) ↔ профиль сопряжения: одна глобальная политика или только пользовательский оверлей.
- **§7:** привязка **стратегического** профиля к конкретному **TopLevel** / `surface_id` + политика при отключённом третьем мониторе (fallback на вкладку / скрытие).
