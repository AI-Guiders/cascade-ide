## 2. Философия продукта: зачем не «вторая VS/Rider» в смысле UX

Классический **Visual Studio** (и близкие по духу IDE) долго были ориентиром **DX** для .NET: единая экосистема, предсказуемые действия, инструменты по делу. Это близко к образу **«хорошего актёра»**: в фокусе задача и код, а не реклама среды.

Другой класс риска — когда в тот же контур **встраивается облачный inline-ассистент по умолчанию везде**: подсказки конкурируют с кодом за внимание, сложно предсказать, что окажется в буфере, страдает **суверенитет** (отключение, граница с облаком). В такой картине IDE становится **«плохим актёром»** — выходит на передний план вместо задачи.

**CascadeIDE** сознательно держит другую планку: **суверенитет, локальность, прозрачность, невидимость по умолчанию** для всего, что конкурирует с кодом; наблюдаемые протоколы (MCP), ADR как канон, одна модель команд — не «магия в каждом символе». Авиационная метафора (кокпит, Dark Cockpit, порядок обхода) — это **язык про дисциплину внимания**, не косплей кабины.
