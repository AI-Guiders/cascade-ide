# ADR 0017: Несколько окон приложения (мультиоконность), зоны экрана и поверхности агента

**Статус:** Proposed (черновик для обсуждения; детали реализации и сроки — после согласования)  
**Дата:** 2026-04-05  
**Связь:** [0010](0010-ui-modes-toml-configuration.md) (режимы и TOML поверх каркаса окна), [0012](0012-floating-workspace-chrome.md) (плавающий хром; отдельные окна как одна из механик), [0016](0016-agent-client-protocol-external-agent.md) (внешний агент по ACP — ортогонально тому, *в скольких окнах* показывается встроенный UI), [0002](0002-debug-human-agent-parity.md) (паритет MCP), [0008](0008-mcp-contracts-and-testable-infrastructure.md) (контракты MCP). UX-метафора зон внимания: [`concept-pfd-mfd-cascade-v1.md`](../ux/concept-pfd-mfd-cascade-v1.md). Режим полигона: `UiModes/Flight.toml`, семья **Flight**.

## Контекст

Сейчас **одно главное окно** (`MainWindow`) задаёт весь видимый кокпит: редактор, дерево, чат, нижний док и т.д. Конфигурация режимов ([0010](0010-ui-modes-toml-configuration.md)) описывает **видимость и метрики поверх фиксированного каркаса**; альтернативная «схема окна» целиком в данных явно отложена.

Потребность: **разнести роли по экрану** без обязательной ужимки всего в одну колонку — в том числе:

- **Несколько физических мониторов** или **один широкий/изогнутый дисплей** с явными зонами (см. концепт PFD/MFD, §6).
- **Чат и тяжёлые панели агента** (история, trace, операции) как кандидаты на **второе окно** или зону, чтобы не конкурировать с редактором за постоянный фокус.
- **Эксперименты с раскладкой** изолированно от стабильных пресетов — режим **Flight** как песочница; мультиоконность логично отрабатывать там первыми (см. §5 `concept-pfd-mfd-cascade-v1.md`).

[0012](0012-floating-workspace-chrome.md) уже допускает **отдельные окна** для части хрома (телеметрия, полоски) и фиксирует, что снимок MCP и действия по UI сейчас завязаны на **один** визуальный корень. Этот ADR **расширяет предмет**: не только плавающий хром, а **продуктовая модель нескольких окон приложения** (какие поверхности, жизненный цикл, состояние, связь с режимами и агентом).

## Решение (принципы; детализация — по итогам обсуждения)

1. **Один процесс, несколько `TopLevel`.** Окна принадлежат одному экземпляру IDE; общее состояние решения, агента, настроек — **единый** граф VM/сервисов за фасадом, без второго «независимого» экземпляра приложения. *Исключения* (если когда-нибудь понадобятся) — отдельное решение и отдельный ADR.

2. **Главный фрейм.** Сохраняется понятие **основного окна** (редактор, решение, главная навигация по умолчанию). Дочерние/вспомогательные окна **не** обязаны дублировать полный кокпит; их состав определяется пресетом режима и/или действием пользователя («вынести чат»). Закрытие главного окна — **политика выхода из приложения** (закрыть все дочерние или запрос подтверждения) — фиксируется в реализации, не оставлять неявным.

3. **Что может жить во втором и следующих окнах (кандидаты, не обязательно всё в v1):** панель чата и связанный MFD-контент агента; компактная PFD-полоса (подтверждения, критичные индикаторы) на отдельном мониторе; выносимые фрагменты хрома в смысле [0012](0012-floating-workspace-chrome.md). **Документы редактора как floating MDI** остаются **вне объёма этого ADR**, пока не будет отдельного решения (как в [0012](0012-floating-workspace-chrome.md)).

4. **Связь с режимами UI ([0010](0010-ui-modes-toml-configuration.md)).** Пресеты задают **видимость и слоты** поверх доступных поверхностей: одно окно или несколько. Расширение: при необходимости ключи/секция уровня «какие панели во вспомогательном окне по умолчанию» — **в TOML режима** или во **Flight**-специфичном файле, без дублирования глобальной схемы `MainGrid` в каждом режиме там, где достаточно флагов.

5. **Сохранение геометрии.** Позиции и размеры вспомогательных окон — **не** обратная запись в шипнутые `UiModes/*.toml` (правило [0010](0010-ui-modes-toml-configuration.md)); целевой канал — **пользовательские настройки** / отдельный блок состояния workspace (как только появится договорённость с существующим `settings.toml` и `workspace.toml`).

6. **Агент (ACP, [0016](0016-agent-client-protocol-external-agent.md)).** Транспорт к внешнему агенту **не зависит** от числа окон: один канал stdio/сессия. Решение этого ADR — только **куда монтируется** встроенный UI ответов и подтверждений (главное окно vs дополнительное). Краткие подтверждения и критичные сигналы по-прежнему должны быть доступны в **PFD-зоне** внимания (концепт §3); при втором окне — либо дубль компактной полосы, либо явное правило «фокус на главном окне для подтверждения».

7. **MCP и паритет ([0002](0002-debug-human-agent-parity.md), [0008](0008-mcp-contracts-and-testable-infrastructure.md)).** Как только в продукте появляется значимый UI **вне** дерева `MainWindow`, **расширение контракта** снимка лэйаута и действий (несколько корней `TopLevel`, идентификация окна/региона) входит **в scope** поставки вместе с фичей — по аналогии с [0012](0012-floating-workspace-chrome.md). До согласования контракта мультиоконные поверхности не должны быть «только для человека».

## Последствия

- Потребуется **дизайн владения VM**: общие сервисы, дочерние окна как представления или отдельные лёгкие VM с подпиской на общий сессионный слой.
- Тесты UI и сценарии с **одним корнем** потребуют обновления или двойного режима (одно окно / два окна).
- Документация для пользователя (позже): как вынести панель, как вернуть, где сохраняется геометрия.

## Отклонённые альтернативы (как целевое состояние)

- **Полагаться только на внешний Cursor/IDE на втором мониторе** без встроенной мультиоконности — отклонено как не закрывающее сценарий «одна Cascade + зоны экрана» и PFD/MFD внутри продукта (см. концепт §8 — внешние инструменты **дополняют**, не заменяют осмысленную раскладку Cascade).
- **Сразу вводить полный MDI документов** — вне объёма; см. [0012](0012-floating-workspace-chrome.md).

## Открытые вопросы (для обсуждения перед кодом)

- Минимальный **v1**: одно вспомогательное окно только под чат или также под нижний док / телеметрию?
- **Z-order и фокус:** как гарантировать видимость модальных подтверждений агента при чате во втором окне.
- Нужна ли **привязка пресета** к конфигурации мониторов (имя/ид дисплея) или достаточно сохранённых прямоугольников.
- Синхронизация с **Flight**: все эксперименты мультиоконности только в Flight до стабилизации — или допускается Power с флагом.
